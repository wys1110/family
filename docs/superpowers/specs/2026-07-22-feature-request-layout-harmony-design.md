# 요청 탭 레이아웃 조화 개선 설계

## 목표

요청 작성과 관리자 요청 목록을 작은 모바일에서는 읽기 쉽고 조작하기 편하게, 데스크톱에서는 과도하게 늘어나지 않는 하나의 일관된 작업 화면으로 정리한다.

실제 로그인 화면에서 확인한 문제는 다음과 같다.

- 1280px 뷰포트에서 외곽 카드가 1,064px, 요청 항목이 1,006px, 상태 선택창이 935px까지 늘어난다.
- 390px 뷰포트에서 상태 행 272px 중 선택창이 244px를 차지해 `상태`가 두 줄로 세로처럼 깨진다.
- 상태 선택창 높이가 모바일 34px, 데스크톱 40px로 최소 터치 영역 44px보다 작다.
- 모바일은 요청 뷰와 앱 셸이 각각 112px, 데스크톱은 각각 136px와 144px의 하단 여백을 적용해 빈 공간이 중복된다.
- 데스크톱 요청 폼에서 textarea 899px와 버튼 95px가 한 행의 양 끝으로 떨어져 시각적 연결이 약하다.
- 별빛 밤에서 작성 카드, 관리자 외곽, 관리자 항목이 서로 다른 표면 깊이를 사용한다.

## 범위

- `feature-request.css`의 요청 화면 레이아웃과 조작 영역을 정리한다.
- `night-feature-request-polish.css`의 별빛 밤 표면 색상을 정리한다.
- 필요한 캐시 버전과 CSS 계약 테스트를 갱신한다.
- 요청 생성, 조회, 상태 저장, Supabase, 권한 로직은 변경하지 않는다.
- SUIT Variable 단일 텍스트 폰트 정책을 유지한다.
- `HANDOFF.md`와 `.superpowers/`는 수정하거나 커밋하지 않는다.
- 승인된 `다가오는 일정` 설계는 보존하되 이번 작업에는 구현하지 않는다.

## 선택한 접근

기존 DOM과 기능을 유지하면서 요청 전용 CSS의 구조를 바로잡는다. 화면별 새 임시 CSS 파일을 추가하거나 전체 디자인 시스템을 다시 만드는 대신, 문제를 만든 원래 선택자에서 폭·grid·padding·표면 토큰을 수정한다.

대안으로 요청 화면 전체를 새 마크업으로 재작성하거나 관리자 목록을 테이블로 바꿀 수 있지만, 전자는 기능 회귀 위험이 크고 후자는 모바일 가독성이 떨어져 채택하지 않는다.

## 레이아웃

### 화면 폭

- `.feature-request-view`의 최대 폭은 880px로 제한하고 중앙 정렬한다.
- 880px 미만에서는 부모의 사용 가능한 폭을 100% 사용한다.
- 작성 카드와 관리자 카드는 동일한 폭과 좌우 기준선을 사용한다.
- 데스크톱에서도 내부 요청 항목은 외곽 카드 폭을 넘지 않는다.

### 요청 작성 폼

- textarea는 모든 해상도에서 한 행 전체를 사용한다.
- textarea 아래에 글자 수를 왼쪽, `요청 등록` 버튼을 오른쪽에 둔다.
- 버튼은 최소 높이 48px, 최소 폭 112px로 한다.
- textarea는 모바일 최소 120px, 데스크톱 최소 132px로 한다.
- 안내 문구는 다음 행 전체를 사용한다.

### 관리자 목록

- 관리자 헤더는 제목·개수와 새로고침 버튼을 양쪽에 배치한다.
- 각 요청 항목의 작성자·시간은 같은 행에서 양쪽 정렬하되 좁은 화면에서는 자연스럽게 줄바꿈한다.
- 요청 본문은 최대 `68ch`로 제한해 데스크톱에서 한 줄이 지나치게 길어지지 않게 한다.

### 상태 행

- 상태 행은 `grid-template-columns: auto minmax(0, 220px)`를 사용한다.
- 모바일에서도 `상태` 라벨이 한 줄을 유지하도록 `white-space: nowrap`을 적용한다.
- 선택창은 사용 가능한 공간 안에서 최대 220px, 최소 높이 44px를 사용한다.
- 모바일 390px에서는 라벨과 선택창을 합쳐 카드 내부 100%를 넘지 않는다.

### 하단 여백

- `.feature-request-view`의 별도 하단 패딩은 제거한다.
- 고정 새로고침 버튼을 위한 공간은 전역 `.app-shell` 하단 패딩 하나만 사용한다.
- 모바일과 데스크톱 모두 마지막 관리자 카드 아래에 전역 패딩만 남긴다.

## 타이포그래피

- 페이지 제목 24px/700, 섹션 제목 18px/700의 기존 공통 토큰을 유지한다.
- 본문과 입력값은 15px, 라벨과 메타 정보는 13px을 유지한다.
- 버튼은 14px/600을 유지한다.
- `상태`와 날짜·작성자 메타는 행간 1.4 이상을 사용한다.
- 기존 장식용 `?` 문자와 Georgia 지정은 제거하고, 텍스트 글리프 없는 CSS 그라데이션 장식으로 대체한다.

## 색상과 표면

### 별빛 밤

- 작성 카드와 관리자 외곽은 동일한 짙은 네이비 표면 계열을 사용한다.
- 관리자 요청 항목은 외곽보다 한 단계 밝은 블루 네이비로 구분한다.
- 상태별 왼쪽 강조선은 현재 의미 색상을 유지한다.
- textarea와 select는 동일한 입력 표면·테두리 대비를 사용한다.
- 등록 버튼은 기존 테마 accent를 유지하되 텍스트 대비를 유지한다.

### 포근한 숲

- 작성 카드와 관리자 외곽은 기존 크림 표면 토큰을 공유한다.
- 관리자 요청 항목은 외곽보다 한 단계 밝은 표면을 사용한다.
- 녹색 accent와 상태별 강조선의 의미는 변경하지 않는다.

## 접근성

- textarea, 등록 버튼, 새로고침, 상태 선택창의 기존 접근 가능한 이름을 유지한다.
- 모든 조작 영역은 최소 44×44px을 만족한다.
- 기존 `focus-visible` outline을 유지한다.
- 상태 라벨과 select 연결을 유지한다.
- 문서와 카드에 가로 스크롤이 생기지 않게 한다.

## 테스트와 실제 검증

- 요청 뷰 최대 폭 880px 및 중앙 정렬 CSS 계약
- 요청 폼의 단일 열 textarea와 하단 action row 계약
- 상태 행 grid, `white-space: nowrap`, select 최대 폭 220px, 최소 높이 44px 계약
- 요청 뷰 중복 하단 패딩 제거 계약
- Georgia 및 장식용 `?` 텍스트 제거 계약
- `npm test`, `npm run check`, `git diff --check`
- 별빛 밤과 포근한 숲에서 390×844, 430×932, 768×1024, 1440×900 실제 로그인 화면 확인
- 각 해상도에서 카드·textarea·상태 선택창 computed width와 font 값을 확인
- `documentElement.scrollWidth <= documentElement.clientWidth` 확인
